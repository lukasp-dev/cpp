#### Memory Bifurcation: Stack v. Heap
When you execute `Order* ptr = new Order();`, the memory is managed in two distinct regions:

- **The Stack (The identifier)**: The pointer variable `ptr` itself is stored on the **Stack**. On a 64-bit system, this occupies **8 bytes** of memory. If serves as a handle that holds the memory address of the actual object.
- **The Heap (The Instance)**: The `new Order()` expression allocates space in the **Heap**for the actual data (the instance). This is where the object's member variables and internal structures reside.
 - **The Connection**: The stack-based **ptr** contains the **hexadecimal memory address** (e.g., `0x7ffee1234568`) that points to the starting byte of the object in the heap.

The separation between Stack and Heap allows C++ to balance automatic speed with manual persistence.

- The Stack (Identifier): Automatic Lifetime
    - **Mechanism:** Governed by Automatic Storage Duration. When a function exits, the "Stack Frame" is popped and the identifier ptr is instantly destroyed.

    - **Behavior:** Local, fast, and deterministic. It "dies" as soon as it goes out of scope.

- The Heap (Instance): Dynamic Lifetime
    - **Mechanism:** Governed by Dynamic Storage Duration. Objects created via new exist independently of the function's execution flow.
    - **Behavior:** Persistent. It remains "alive" until delete is explicitly called.

#### The Memory Leak Trap
A Memory Leak occurs when the "bridge" between these two is broken:
1. The function exists, and the **Stack-based identifier** (`ptr`) is destroyed.
2. The **Heap-based instance** remains alive, but its address is now lost.
3. The memory becomes **unreachable**: it occupies RAM but can never be reclaimed or reused.
